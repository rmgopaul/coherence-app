## 🐛 Bugs found

### P0 — production-impacting

#### Bug #1 — 502 Bad Gateway cascade under heap pressure
**Repro:** Click "Snapshot Log" tab → ~30 sequential `getDashboardSystemsPage` calls fire to walk all 24,293 systems. After the first ~20 succeed, the next 8 (requests 21-28) return **502 Bad Gateway**. Client retries and eventually completes (requests 29+ return 200). On a subsequent tab navigation to Trends, 4 procs (`getDashboardTrendDeliveryPace`, `getDashboardTrendsProduction`, `getScopeId`, `listDatasetUploadJobs`) returned **persistent 502s** that did NOT auto-retry — user is stuck on broken tab until manual refresh.

**Likely root cause:** Express request handler returns `TRPCError({ code: "TOO_MANY_REQUESTS" })` from `dashboardResponseGuard.ts`'s pre-flight `DASHBOARD_HEAP_PRESSURE_REJECT_BYTES` check (defaults to 2 GB). Render's load balancer sees the error response and translates to 502 at the LB layer rather than passing the 429 through. Hard refresh recovers the worker (heap GC + fresh page load).

**Fix candidates:**
1. Make `dashboardResponseGuard` return a clean 429 with a `Retry-After` header instead of throwing TRPCError, so Render LB passes it through cleanly
2. Have the tRPC client auto-retry on 502 with exponential backoff
3. Reduce the trigger — see Bug #5 (Snapshot Log fires 30+ paginated calls in sequence, the immediate cause of the heap pressure)

**Severity:** P0 — sustained user-facing 502s during normal navigation flow.

### P1 — wrong data shown to users / unexpected behavior

#### Bug #2 — URL `?tab=...` deep-link doesn't activate the target tab on cold mount
**Repro:** Navigate directly to `https://app.coherence-rmg.com/solar-rec/dashboard?tab=application-pipeline`. URL stays as `?tab=application-pipeline` (correct), but the rendered content is the **Overview tab** (Total Systems / Reporting / size buckets / financial tiles). Bookmarks and shared deep-links to specific tabs always land on Overview.

**Likely root cause:** The active-tab state is initialized from local component state (e.g. `useState("overview")`) rather than read from `searchParams.get("tab")` on mount. The tab buttons update both URL and state when clicked, but the URL-on-mount path is missing.

**Fix:** In the dashboard's tab-state hook, initialize `activeTab` from the URL query param if present.

**Severity:** P1 — broken deep-linking; user can still reach tabs by clicking, but bookmarks / shared links / browser back navigation are broken.

#### Bug #3 — Delivery Tracker fires 11 concurrent `applyScheduleBToDeliveryObligations` mutations on activation
**Repro:** Click "Delivery Tracker" tab → network panel shows 11 concurrent `applyScheduleBToDeliveryObligations` POSTs, all in `pending` status simultaneously, none completing in the observation window.

**Likely root cause:** Per CLAUDE.md, this proc is part of the Phase 5d Salvage PR C — it writes Schedule B → delivery obligations and is supposed to be **30-second-throttled**. The throttle guard is bypassed somehow on tab activation. Could be a `useEffect` dep array that includes a freshly-recreated reference each render, causing fires-on-every-render until the throttle ref settles.

**Risk:** Each call is a WRITE to `srDsDeliverySchedule` rows. 11 concurrent writes per tab activation may corrupt the dataset (last-writer-wins, race conditions, duplicate transitions).

**Severity:** P1 — wrong data writes happening on every Delivery Tracker visit.

#### Bug #4 — Snapshot Log header banner shows wrong financial values during paginated load (Flash Of Wrong Data)
**Repro:** Click "Snapshot Log" tab. Header banner "Total Contracted Value (Part II Verified)" cycles through:
- $67,008,912.09 (during first ~5s)
- $253,582,030.15 (during ~10-15s as pages stream in)
- $478,105,589.58 (final, ~20s)

Same label, same banner — but the value displayed shifts substantially as the paginated systems-walk progresses. User briefly sees inaccurate portfolio totals during this window.

**Likely root cause:** The header tile reads from a client-side aggregation across `getDashboardSystemsPage` results. Each new page added recomputes the sum. During the load window, the displayed value reflects a partial sum.

**Fix:** Either (a) pull the financial header values from `getDashboardFinancialKpiSummary` (server-side aggregate, instant) instead of recomputing client-side, or (b) suppress the header until all pages have loaded.

**Severity:** P1 — user sees wrong numbers during normal navigation. Financial values especially sensitive.

#### Bug #5 — Performance Ratio Allocation Detail has duplicate rows
**Repro:** Click "Performance Ratio" tab → scroll to "Performance Ratio Allocation Detail" table. Page 1 shows the system "Rob Horton" twice as separate rows:

| | System | NONID | Portal ID | Monitoring | Match Type | Read Date | Lifetime Read | Delta | Expected | Ratio | Contract |
|---|---|---|---|---|---|---|---|---|---|---|---|
| Row 1 | Rob Horton | NON564967 | 107458 | SolarEdge | Monitoring + System Name | Apr 13, 2026 | 71,081.48 | +54,240.48 | 14.63 | 71,510.14% | $5,443.75 |
| Row 2 | Rob Horton | NON564967 | 107458 | SolarEdge | Monitoring + System Name | Apr 13, 2026 | 71,081.48 | +54,240.48 | 14.63 | 71,510.14% | $5,443.75 |

Identical NONID, Portal ID, monitoring, match type, lifetime read, delta, expected, ratio, contract. True duplicate — not a multi-match-type spread.

**Likely root cause:** The parent `solarRecDashboardPerformanceRatioFacts` PK is `(scopeId, buildId, key)` where `key = ${convertedReadKey}-${candidateKey}`. A true duplicate requires either (a) the matcher emitting two distinct `convertedReadKey`s for the same source row, or (b) two separate fact rows being upserted with different `key` values that nonetheless render identically.

**Recommended diagnostic:** Query prod TiDB:
```sql
SELECT key, convertedReadKey, COUNT(*)
FROM solarRecDashboardPerformanceRatioFacts
WHERE scopeId = 'scope-user-1'
  AND systemId = '107458'
  AND readDate = '2026-04-13'
  AND matchType = 'Monitoring + System Name'
GROUP BY key, convertedReadKey;
```

**Severity:** P1 — user-visible data integrity issue (every fact-table consumer counts these as 2 separate allocations).

## Deep-dive diagnoses + fix plan (added 2026-05-09)

After a follow-up code-walk through the relevant files, here is the
finalized root cause + fix shape for each bug, packaged into 7 focused
PRs.

### PR-1 — Bug #5 (Performance Ratio Allocation Detail duplicates)

**Root cause (confirmed by code-walk):** `convertedReadsBridge.ts`
maintains separate sources `mon_batch_<provider>` (API push) and
`individual_<provider>` (manual CSV upload). Per-source dedup uses
`convertedReadsRowKey = monitoring|systemId|systemName|lifetime|date`.
Cross-source dedup does NOT happen — when both sources contain the
same physical reading via different ID populations (one with
`monitoring_system_id="1206921"`, one with empty), they have
different per-source keys and both rows survive into
`srDsConvertedReads`.

The matcher in `buildPerformanceRatioAggregates.ts:177–316` then
emits one `PerformanceRatioRow` per `(globalIndex, candidateKey)`
tuple. Both source rows for "Rob Horton" match the SAME system via
"Monitoring + System Name" priority → two identical fact rows,
identical match type, identical lifetime, identical delta.

**Fix shape:** Pre-emission dedup in the matcher accumulator. Track a
`processedCrossSourceKeys: Set<string>` keyed by
`monitoring|sysName|lifetime|readDate` (sysId omitted) across pages.
When a row's cross-source key already appeared, skip emission.
Counter accounting: skipped rows count as `matchedConvertedReads`
(they DID find a match) but don't emit a duplicate fact.

**File:** `server/services/solar/buildPerformanceRatioAggregates.ts`
**LOC est:** ~30 prod + ~80 test

### PR-2 — Bug #2 (URL `?tab=` deep-link)

**Root cause:** The lazy `useState` initializer at
`SolarRecDashboard.tsx:1998–2000` reads from `useSearch()` and falls
back to `DEFAULT_DASHBOARD_TAB` when `getTabFromSearch` returns null.
The user's repro URL was `?tab=application-pipeline` but the
canonical slug is `app-pipeline` (see
`client/src/solar-rec-dashboard/lib/constants.ts:193–214`). The
verbose form falls through validation → user lands on Overview.

**Fix shape:** Add a small alias map at `getTabFromSearch` so the
intuitive verbose form maps to the canonical short form. Cover at
minimum `application-pipeline` → `app-pipeline`. Fix the
useState initializer to use `window.location.search` directly (more
resilient than `useSearch()` on cold mount with a base path). Add
unit tests covering both forms + invalid slug fallthrough.

**File:** `client/src/features/solar-rec/SolarRecDashboard.tsx`
(lines 497–502, 1998–2000) + new test file
`client/src/features/solar-rec/getTabFromSearch.test.ts`
**LOC est:** ~20 prod + ~40 test

### PR-3 — Bug #3 (Schedule B mutation fan-out)

**Root cause:** `ScheduleBImport.tsx:530–614` has the auto-apply
`useEffect`. Its dep array (lines 608–613) includes
`applyScheduleBToDeliveryObligations` — the result of
`useMutation()`. **`useMutation()` returns a new object reference on
every render**, so any parent re-render re-fires this effect. Each
fire schedules a `setTimeout` with `delay = max(0, 30s − elapsed)`;
when `elapsed > 30s`, every fire schedules a 0ms timeout. 11
fires-per-render-burst → 11 concurrent mutateAsync calls.

**Risk:** Each mutation reads + merges the Schedule B → delivery
schedule, then writes the merged set back. 11 concurrent reads of
the same baseline → 11 independent merges → last-writer-wins on
`srDsDeliverySchedule` rows. Data corruption surface.

**Fix shape:** Stabilize the mutation reference. Hoist
`applyScheduleBToDeliveryObligations` into a ref
(`mutationRef.current`); the effect calls `mutationRef.current.
mutateAsync()` and the dep array drops the unstable mutation
reference. Also: track the active timer in
`autoApplyTimerRef` so back-to-back effect fires REPLACE the
pending timer instead of stacking new ones (`clearTimeout` the
previous before scheduling the next).

**Defense in depth:** Server-side per-job semaphore in
`applyScheduleBToDeliveryObligations` to coalesce concurrent
applies — but this is follow-up; PR-3 fixes the proximate cause.

**File:**
`client/src/solar-rec-dashboard/components/ScheduleBImport.tsx`
**LOC est:** ~40 prod + ~80 test

### PR-4 — Bugs #4 + #7 (Snapshot Log header server-aggregate)

**Root cause:** `snapshotPart2ValueSummary` memo
(`SolarRecDashboard.tsx:3411–3435`) live-reduces over
`part2EligibleSystemsForSizeReporting`, which is itself accumulated
across pages of `getDashboardSystemsPage`. As pages stream in, the
header tile `Total Contracted Value (Part II Verified)` rebinds to
the partial sum. User sees $67M → $253M → $478M cycle.

The cross-tab drift (#7) has the same cause: Overview reads from the
server-aggregated `getDashboardSummary` (via `slimSummary`); Snapshot
Log re-computes locally. Two different code paths over slightly
different inputs → $90K of drift over $478M.

**Fix shape:** The server already exposes the right values via
`getDashboardFinancialKpiSummary` (slim, cache-only,
~200 B response). Switch the Snapshot Log header tile binding to
`financialKpiSummary` instead of `snapshotPart2ValueSummary`.
For the Snapshot-Log-only fields the server doesn't expose
(reporting/non-reporting splits of contracted value), gate the
display behind `isPart2EligibleSystemsPagesComplete` so the user
only ever sees the FINAL value (no partial). Loading shimmer until
then.

**File:** `client/src/features/solar-rec/SolarRecDashboard.tsx`
+ `SnapshotLogTab.tsx` if the prop shape changes
**LOC est:** ~50 prod + ~30 test

### PR-5 — Bug #6 (Snapshot Log eager pagination removal)

**Root cause + interaction with PR-4:** The systems walk at
`SolarRecDashboard.tsx:3326–3358` is enabled when ANY of [Size,
Value, Offline Monitoring, Financials, Snapshot Log] is active.
Once PR-4 ships, Snapshot Log no longer needs the walk for the
header — but `SnapshotLogTab.tsx` itself does NOT consume
`part2EligibleSystemsForSizeReporting` (only `snapshotPart2ValueSummary`,
which PR-4 sources from the server). So the walk is fully redundant
for the Snapshot Log surface.

**Fix shape:** Drop `isSnapshotLogTabActive` from the
`isPart2EligibleSystemsNeeded` predicate. The walk no longer fires
on Snapshot Log activation. Other tabs (Size, Value, etc.) keep
firing it as before — they actually consume the rows.

**Order:** Must ship AFTER PR-4 (or in the same PR) so the header
doesn't break when the data source disappears.

**File:** `client/src/features/solar-rec/SolarRecDashboard.tsx`
(lines 3326–3331)
**LOC est:** ~5 prod + ~30 test (regression rail in
`dashboardMountResilience.test.ts` style)

### PR-6 — Bug #1 (502 cascade resilience)

**Root cause (confirmed):**
`server/_core/dashboardResponseGuard.ts:380–393` throws
`TRPCError({ code: "TOO_MANY_REQUESTS" })` on heap pressure. tRPC's
HTTP adapter emits HTTP 429. **Render's load balancer translates
429 → 502 at the LB layer** (non-2xx from origin = "bad gateway").
Client sees 502.

`getDashboardSystemsPage.useInfiniteQuery` is configured with
`retry: false` (in ComparisonsTab.tsx and the parent's
`part2EligibleSystemsPagesQuery` at line 3340 — implicit default
since no retry override). With `retry: false`, requests 21–28 fail
permanently; subsequent tab navigation to Trends triggers fresh
queries that ALSO see 502 from the still-pressured worker, and
without retry they stay broken until manual refresh.

**Fix shape (3 layers):**
1. **Server emits `Retry-After` header.** Even if Render LB
   currently strips it, this is the correct semantic. Future LB
   tuning may pass it through.
2. **Client retry-on-429/502 with bounded exponential backoff.**
   For `getDashboardSystemsPage` and any other dashboard infinite
   queries: `retry: 3, retryDelay: (attempt) => Math.min(30000,
   1500 * 2 ** attempt)`. Skips retry for 4xx that aren't 429.
3. **Reduce trigger** (already covered by PR-5 + PR-4 — Snapshot
   Log walk gone, so the cascade is shorter).

**Files:** `server/_core/dashboardResponseGuard.ts` +
`client/src/solar-rec/solarRecTrpc.ts` (or wherever the QueryClient
defaults live; expecting QueryClient construction in
`solar-rec-main.tsx`) + the per-query overrides on the systems-page
infinite queries.
**LOC est:** ~50 prod + ~80 test

### PR-7 — Bug #8 (Overview count drift)

**Root cause (confirmed):** Slim aggregator
(`buildSlimDashboardSummary.ts`) and heavy aggregator
(`buildOverviewSummaryAggregates.ts`) compute size buckets +
reporting tiles via fundamentally different paths:

- **Slim:** Streams `srDsSolarApplications` rows, dedupes by CSG ID
  first-win, classifies by raw `installedKwAc <= 10`, reads
  `isReporting` from foundation's `canonicalSystemsByCsgId`.
- **Heavy:** Reads pre-computed `snapshot.systems` array, filters by
  pre-derived `sizeBucket` field, reads `isReporting` from snapshot
  + ABP row matching.

Numbers differ by 25–60 per tile (within ε of total volume —
suggests dedup-rule edge cases or stale snapshot fields).

**Fix shape:** Make the OverviewTab use the SAME path for size
buckets + reporting on both cold mount and post-interaction.
Cleanest approach: read these tiles from `getDashboardSummary` (slim)
unconditionally. The heavy `getDashboardOverviewSummary` is still
used for fields slim doesn't expose (ownership tile, etc.) but does
NOT contribute to size/reporting counts. This eliminates the visible
shift.

If slim is wrong (its 17,645 vs heavy's 17,705 implies slim
under-counts by ~60), the underlying data divergence still needs
investigation — that's a follow-up. For PR-7 the user-visible bug is
"the numbers shift while I'm using it"; pinning the source removes
the shift.

**Files:** `client/src/solar-rec-dashboard/components/OverviewTab.tsx`
+ unit test verifying size/reporting tiles read from slim summary
even when heavy overview-summary is loaded.
**LOC est:** ~30 prod + ~50 test
